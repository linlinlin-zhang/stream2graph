# Stream2Graph 项目作战手册（中文）

本文档整理了截至 `2026-03-11` 为止围绕本项目进行的主要技术、研究与投稿决策。  
它不是简单的聊天记录，而是一份可以持续使用和扩写的项目工作手册。

## 1. 项目现状

这个仓库不是单纯的数据集仓库，也不是单纯的模型训练仓库。  
它目前同时包含四层内容：

- 数据集工程
- 算法与实时流水线原型
- 前端与演示界面
- 微调、评测与运维工具

当前最核心的目录角色如下：

- `versions/`
  - 保存历史版本和当前研究主资产，尤其是 `v3` 主线
- `tools/`
  - 保存发布构建、评测、服务、微调、运维等仓库级脚本
- `frontend/`
  - 保存实时演示界面
- `reports/`
  - 保存运行结果、发布结果和实验结果
- `docs/`
  - 保存运维说明、训练说明、工作区说明和项目计划

因此，这个项目应被理解为一个围绕“实时对话成图”目标构建的完整研究系统。

## 3. 数据复杂度总结

当前发布子集并不简单。整体规模属于中等，但长尾复杂样本相当明显。

### 3.1 对话复杂度

基于 `4709` 条发布样本统计：

- 平均对话轮数：`16.35`

- `p90` 对话轮数：`38`
- `p95` 对话轮数：`59`
- 最大对话轮数：`120`

对话文本长度近似统计：

- 平均对话字符数：`657.35`
- `p90` 对话字符数：`1596`
- `p95` 对话字符数：`2650`

### 3.2 Mermaid 复杂度

基于同一批发布样本统计：

- 平均 Mermaid 字符数：`1238.89`
- `p50` Mermaid 字符数：`817`
- `p90` Mermaid 字符数：`2795`
- `p95` Mermaid 字符数：`3755`
- 最大 Mermaid 字符数：`9669`

按非空代码行近似统计：

- 平均代码行数：`31.67`
- `p50` 代码行数：`22`
- `p90` 代码行数：`62`
- `p95` 代码行数：`93`
- 最大代码行数：`261`

按启发式图结构信号近似统计：

- 平均边数量：`10.91`
- `p90` 边数量：`25`
- `p95` 边数量：`40`
- 最大边数量：`413`

- 平均唯一节点 id 数量：`6.43`
- `p90` 唯一节点 id 数量：`16`
- `p95` 唯一节点 id 数量：`27`
- 最大唯一节点 id 数量：`61`

### 3.3 token 长度估计

使用 `Qwen3.5` tokenizer 对发布集随机样本估算：

- 平均整条样本长度约 `845 tokens`
- `p90` 约 `1760 tokens`
- `p95` 约 `2431 tokens`
- 超过 `1024 tokens` 的样本约占 `26.8%`
- 超过 `1536 tokens` 的样本约占 `12.4%`
- 超过 `2048 tokens` 的样本约占 `8.2%`

由此带来的训练含义是：

- `1024` 能跑，但会截断较多长样本
- `1536` 是很实用的折中
- `2048` 如果硬件允许会更理想

- `architecture`: `588`
- `gantt`: `569`

## 4. 模型策略

### 4.1 当前推荐主线

目前推荐的模型策略是：

- 主模型：`Qwen3.5-27B`
- 快速 baseline：`Qwen3.5-9B`
- 可选比较模型：`Qwen3.5-35B-A3B`

这已经替代了更早期的 `Qwen2.5` 主线建议。

### 4.2 为什么主推 `Qwen3.5-27B`

本项目的任务不是简单分类，而更接近：

- 多轮协作对话理解
- 长上下文指令跟随
- 结构化代码生成
- 语法约束下的 Mermaid 输出

从这个角度看，`Qwen3.5-27B` 目前是最适合作为主线学生模型的开源基座，质量和可实施性之间平衡最好。

### 4.3 为什么还要保留 `Qwen3.5-9B`

`Qwen3.5-9B` 仍然值得训练，原因包括：

- 作为低成本微调 baseline
- 作为低延迟部署候选
- 作为后续混合路由系统的快路径
- 作为论文中可解释的模型规模对照点

它不应该取代 `27B` 成为主结果模型，但非常值得保留。

### 4.4 如何看待 `Qwen3.5-35B-A3B`

`Qwen3.5-35B-A3B` 是 `MoE` 模型，而不是可以把 `27B` 调成“只激活 9B”的模型。

因此更实际的结论是：

- 它可以作为 MoE 架构比较线来测试
- 它不适合作为第一主线训练目标
- 更适合作为补充实验、架构对比或附录结果

如果时间有限，应优先完成 `27B` 和 `9B`，再考虑 `35B-A3B`。

### 4.5 为什么使用 `4-bit QLoRA`

当前硬件计划下，`4-bit QLoRA` 是最现实的训练路线，因为它：

- 显著降低显存压力
- 让 `27B` 级模型在单卡租赁场景下变得可行
- 对中等规模任务适配是合理选择

需要明确的是：

- 最终保存下来的 LoRA adapter 很小，可以下载到本地
- 但推理时仍需要能够加载基座模型的算力环境
- 训练和主评测应放在云端完成，而不是依赖本地电脑

## 5. 训练与部署工作流

### 5.1 推荐训练流程

推荐工作流如下：

1. 租用云 GPU
2. 将仓库克隆到云实例的高速本地盘
3. 在云上完成微调
4. 训练结束后立刻在云上跑验证集和测试集
5. 保存以下产物：
   - adapter checkpoint
   - 配置文件
   - 日志
   - 指标结果
   - 预测输出
6. 将这些产物下载回本地长期保存

主模型的推理和部署也应尽量保持在云端。

### 5.2 微调后会得到什么

如果走 `LoRA / QLoRA` 路线，最终最重要的产物通常不是整模型，而是：

- LoRA adapter
- config
- tokenizer 相关文件
- 训练日志
- 验证与测试结果

即使本地机器无法运行完整模型，也可以把这些结果安全保存下来。

### 5.3 微调后怎么实际使用

推荐的使用方式是：

- 在云上训练
- 在云上测试
- 在云上跑主推理服务
- 本地只负责 UI、开发和结果查看

如果后续确实需要更快的服务路径，可以考虑：

- `9B` 作为快路径
- `27B` 作为高质量路径
- 或者构建一个混合路由系统

## 6. 实验策略

### 6.1 已讨论的四类主实验

目前已经明确过的四类实验是：

1. 部分激活模型与全参与模型对比
2. 与通用大模型对比
3. 与非大模型传统算法模式对比
4. 同一模型微调前后对比

### 6.2 实验优先级建议

即使都能做，它们的重要性也并不相同。

#### 必做

- 同一模型微调前后对比
- 最终系统与传统/启发式 baseline 对比
- 少量通用大模型对比

#### 建议做

- 一个近参数量开源 baseline
- 一个强闭源 API baseline

#### 可选

- MoE 与 dense 的架构对比

如果资源紧张，应最先削减架构对比，而不是削减核心系统比较。

### 6.3 最优最小实验包

对于一篇完整但可执行的论文，推荐的最小实验包是：

- `Qwen3.5-27B base` vs `Qwen3.5-27B + SFT`
- 最终系统 vs heuristic baseline
- 最终系统 vs 一个近参数量开源模型
- 最终系统 vs 一个强闭源 API 模型
- 一轮用户研究

只有这些都稳定之后，再考虑扩展更多模型比较。

## 7. 用户研究策略

### 7.1 是否需要做人评

如果论文主张包括以下任何一点，则最好有人类用户证据：

- 系统更有用
- 系统更高效
- 系统改善了实际成图工作流
- 系统减少用户负担

如果没有用户研究，论文主张就必须收缩为更保守的版本，例如：

- 图质量提升
- 编译成功率提升
- 延迟和稳定性改善

### 7.2 用户研究应评测什么

用户研究测的是“系统体验”，不是每一个模型变体。

推荐的三个系统条件：

1. `Manual`
   - 用户手工整理并成图
2. `Heuristic`
   - 当前启发式/传统实时系统
3. `Final System`
   - 最终模型驱动系统

不要在第一次人评里让参与者同时体验过多模型版本。

### 7.3 参与者数量建议

第一轮用户研究建议：

- `12-16` 名参与者

适合的用户群体：

- 软件工程师
- 产品经理
- 研究人员或研究生
- 平时需要做会议整理、需求梳理或图表表达的人

### 7.4 实验设计建议

推荐使用 `within-subject` 设计，并做顺序平衡。

任务建议：

- 总共 `6` 个任务
- `2` 个简单
- `2` 个中等
- `2` 个复杂

每个任务都基于保留测试集中的会议式对话材料，让参与者产出或修订 Mermaid 图。

每位参与者总实验时长建议：

- `60-75` 分钟

### 7.5 需要记录什么

客观指标：

- 任务完成时间
- 修改次数
- 最终图质量
- Mermaid 编译是否成功
- 首次有用图的出现时间
- 接受或拒绝系统建议的次数

主观指标：

- 感知有用性
- 是否造成打扰
- 信任感
- 工作负担
- 可用性

定性材料：

- 短访谈
- 失败案例反馈
- 系统偏好排序

### 7.6 统计分析

推荐的基础分析方式：

- 小样本下优先使用配对的非参数检验
- 多重比较做校正
- 访谈做轻量主题分析

目标不是把人评做得极大，而是做得规范、可信、能支撑论文主张。

## 8. 会议策略

### 8.1 当前最推荐的会议

当前首推投稿目标：

- `ICMI 2026`

理由：

- 与多模态交互系统、对话、数据集都很契合
- 时间上比 `UIST 2026` 更现实
- 除非主动将项目转成 accessibility 方向，否则比 `ASSETS 2026` 更适合

### 8.2 为什么不优先投 `UIST 2026`

`UIST` 虽然在交互方向上很契合，但它更要求：

- 交互故事足够打磨
- 系统展示质量极高
- 时间节奏更紧

按当前时间点来看，风险偏高。

### 8.3 为什么不优先投 `ASSETS 2026`

只有在项目明确转向无障碍/辅助场景，并且用代表性目标用户完成验证时，`ASSETS` 才会成为更合适的选择。

否则，它不应作为当前第一目标。

### 8.4 如果以 ICMI 为主投目标，应如何调整

论文要突出以下内容：

- 多模态或协作式对话理解任务
- 数据集贡献
- 实时交互系统贡献
- 集中而可信的评测
- 一轮扎实的小规模用户研究

不要把论文做成纯模型排行榜。

## 9. 团队配置与分工

当前假定的团队配置是：

- 2 人负责技术
- 2 人负责用户研究
- 1 位老师作为通讯作者/方向把关

这对 `ICMI` 风格的论文来说是完全合理的结构。

建议分工如下：

- 技术负责人 A
  - 训练
  - 推理
  - 离线评测
- 技术负责人 B
  - 实时系统接入
  - 日志记录
  - demo 稳定性
- 用户研究负责人 A
  - 实验设计
  - 任务材料
  - 问卷
- 用户研究负责人 B
  - 招募
  - 实验执行
  - 定性分析
- 老师
  - 收敛论文主张
  - 审核研究方法
  - 推进写作节奏

## 10. 如何控制项目范围

即使时间允许，也不应对所有实验投入相同精力。

推荐投入顺序：

1. 主模型与微调
2. 微调前后对比
3. 传统 baseline 对比
4. 一个强开源或强闭源外部 baseline
5. 用户研究
6. 可选的 MoE vs dense 架构比较

如果必须砍内容，应先砍模型广度，再砍用户研究质量。

## 11. 当前推荐的下一步动作

建议按如下顺序推进：

1. 确认主投会议为 `ICMI 2026`
2. 确认主模型为 `Qwen3.5-27B`
3. 保留 `Qwen3.5-9B` 作为次级 baseline
4. 暂缓 `Qwen3.5-35B-A3B`，除非有额外时间
5. 冻结评测协议
6. 冻结用户研究 protocol
7. 在云上跑第一版主模型训练
8. 完成最小可用比较实验
9. 完成用户研究
10. 围绕一个清晰主张完成论文写作

## 12. 当前最清晰的论文主张候选

推荐的核心主张是：

`A fine-tuned dialogue-to-diagram model, coupled with a realtime incremental rendering pipeline, produces more accurate and more usable Mermaid diagrams from collaborative dialogue than prompting-only general models and heuristic baselines.`

对应的中文含义可以表述为：

`基于高质量对话成图数据微调得到的模型，配合实时增量渲染流水线，能够比仅依赖提示词的通用大模型和启发式基线系统生成更准确、也更可用的协作式 Mermaid 图。`

这条主张足够集中，能够同时支撑模型实验、系统实验和用户研究。
